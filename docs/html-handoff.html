Runtime Map

VPS routelet
Controlled Chromium renders the overlay. Xvfb captures video. Pulse/PipeWire captures browser audio. FFmpeg maps 0:v:0 and 1:a:0 into Twitch RTMP.
Twitch scaling
Under 10 active chatters: @personality messages queue with a roughly 3s gap. Above 10: messages batch into room-level replies.
OpenAI state keys
Twitch state uses twitch:<channel>:persona:<name>. Local chat uses local:persona:<name>. Memory refreshes run stateless.
POML prompt renderer
POML now sits between app state and OpenAI payloads. The browser calls the bot API render endpoint, which uses the fixed pomljs tarball before chat requests.
Client speech
Piper remains browser-side. LLM deltas accumulate into punctuation chunks, then TTS starts on chunks while later text is still arriving.
Motion truth
Direct DiP BVH/VRMA exports are retired. Keep SMPL cache and use VMC/tracker streaming or a real humanoid retarget before app-facing clips.
Current animation pack
Silly Tavern BVHs are imported into the sequencer with purpose/tag classification. Tune weights before enabling chaotic clips in normal idle loops.

POML Integration

Role

  • Use src/lib/chat/templates/yourwifey-responses.poml as the prompt/context renderer for OpenAI Responses API requests.
  • Bridge it through src/lib/chat/poml.ts, then call it from src/lib/chat/prompt.ts.
  • Render with server/src/ai/PomlRenderer.ts via /ai/poml/render; dependency is pinned to vendor/pomljs-0.0.9.tgz.
  • Keep app code responsible for live state writes, memory commits, TTS, and animation selection.
  • Render the same request shape for local chat and Twitch chat, with different state keys.
  • Keep memory refresh/summarization prompts stateless so they cannot pollute live conversation state.

Implementation status

  • Current app integration is intentionally narrow: a checked-in POML-shaped template plus src/lib/chat/poml.ts.
  • This avoids adding an unstable upstream parser dependency to the stream runtime while keeping the prompt layer portable to real POML.
  • Tests live in src/lib/chat/prompt.test.ts and cover template discovery, context insertion, metadata instruction insertion, and Responses instructions/input splitting.
  • If the app later imports the upstream POML package directly, use the fork/PR that fixes raw </&, multiline expressions, fallback variables, <if condition>, and <tool-policy>.

Dynamic data and caching

  • Send instructions every turn. Responses previous_response_id does not carry old instructions forward, and fresh persona/memory/time data must be visible to the current turn.
  • Keep stable policy, reply metadata contract, and persona text at the top of the rendered prompt so OpenAI prompt caching can hit the repeated prefix.
  • Put dynamic turn metadata, channel facts, active chatter counts, time, retrieved memory, and host context after the stable prefix.
  • In Conversation mode, changing dynamic instructions must not create a new conversation id. The state key still owns isolation: twitch:<channel>:persona:<name>.

Inputs

  • Active persona profile: name, aliases, voice, background, VRM, behavior rules.
  • Conversation scope: Twitch channel/persona key or local persona key.
  • Relevant relationship memory and semantic memory snippets.
  • Recent chat context, active chatter mode, controller/admin facts, and current turn.
  • Animation catalog with safe tags like idle, talking, laugh, thinking, greeting, and react.

Outputs

{
  "instructions": "persona, memory-use rules, tool policy, response contract",
  "input": [
    { "role": "user", "content": "current chat turn" }
  ],
  "text": {
    "format": {
      "type": "json_schema",
      "name": "yourwifey_reply",
      "strict": true
    }
  }
}

Reply contract

  • speech: short spoken reply for streaming TTS.
  • subtitle: optional cleaned display text when different from speech.
  • emotion: stable mood label for expression and animation weighting.
  • motion: optional safe animation intent, not a raw file path.
  • memory: optional candidate facts for the browser DB/memory worker to validate.

Architecture Notes

Stream flow

Twitch chat -> browser IRC -> OpenAI proxy -> browser Piper TTS
                                            -> VRM lipsync/subtitles

Chromium page video -> Xvfb display -> FFmpeg input 0
Chromium/Piper audio -> Pulse/PipeWire sink -> FFmpeg input 1

FFmpeg -map 0:v:0 -map 1:a:0 -> Twitch RTMP

Motion bridge

DiP / MDM motion
-> SMPL params
-> tracker transforms or humanoid retarget
-> VMC/VMT live control or VRMA/FBX baked clip
-> app sequencer only after visual validation
